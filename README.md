# Intake Orders Service

Сервис не про "ещё один CRUD-магазин", а про грязную зону между приёмом заказа и складом.

Идея простая: API принимает заказ, нормализует позиции, режет откровенно плохие кейсы на входе и кладёт заявку в Redis-очередь. Worker дальше решает, что с заказом делать:

- отдать в обработку сразу;
- отложить и попробовать позже, если внешний фид склада ещё не готов;
- отправить в `manual_review`, если заказ выглядит как рискованный для автопотока;
- завершить в `failed`, если кейс уже точно не починится ретраями.

## Почему это выглядит как рабочий сервис, а не как туториал

- на входе схлопываются дублирующиеся позиции;
- сумма пересчитывается и сверяется с тем, что прислал клиент;
- крупные и хрупкие партии не проходят по счастливому пути, а падают в ручную проверку;
- retry не мгновенный, а через отложенную Redis sorted set;
- у сервиса есть понятный intake-сценарий, а не абстрактное "что-то обрабатываем асинхронно".

## Статусы заказа

- `queued` — заказ принят и ждёт worker;
- `processing` — worker взял заказ в работу;
- `done` — заказ прошёл автоматический intake;
- `failed` — заказ нельзя провести автоматически;
- `manual_review` — заказ должен посмотреть оператор.

## Бизнес-правила

### На входе API

- пустой заказ запрещён;
- позиции с одинаковым названием схлопываются в одну строку;
- у одной и той же позиции не может быть разных цен;
- слишком раздробленные заказы режутся на входе;
- заявленный `total` должен совпадать с суммой строк.

### В worker

- заказы с `preorder-*` считаются временно неподтверждёнными и уходят в delayed retry;
- заказы с `blocked-*` считаются нерешаемыми автоматически и завершаются ошибкой;
- крупные партии и хрупкие пачки уходят в `manual_review`.

## Стек

- FastAPI
- PostgreSQL
- Redis как очередь, delayed retry и lock-хранилище
- SQLAlchemy async
- Alembic

RabbitMQ убран намеренно: для этого сценария Redis даёт более честный и компактный operational footprint.

## Пример запроса

```json
{
  "customer": "ops@northwind.example",
  "items": [
    {"name": "book", "price": 100, "qty": 1},
    {"name": " book ", "price": 100, "qty": 2},
    {"name": "preorder-marker", "price": 40, "qty": 1}
  ],
  "total": 340
}
```

Что произойдёт:

1. API схлопнет две строки `book` в одну;
2. пересчитает итог;
3. положит заказ в `orders:ready`;
4. worker увидит `preorder-marker` и переведёт заказ в отложенный retry.
